Manage Passive Sensors via Nidstool

This section describes the nidstool utility for eyeInspect Passive Sensors.

Nidstool is a management utility for eyeInspect Passive Sensors. It can be used to perform activities such as checking the status of the sensor containers, starting/stopping/restarting them, and so on.

When installing Passive Sensors with the deployer, nidstool will also be installed alongside.

The following table describes the various nidstool commands to manage passive sensors.

Nidstool commands are executed from the Linux terminal, from any location on the system. For an overview of the supported commands, execute the following command:

sudo nidstool --help

The following table lists the available nidstool commands.
CommandSyntaxDescriptionExample
LS
sudo nidstool ls

Prints the list of known sensors services. It will print the name, the docker container status, interfaces and uptime of each sensor.

 
UP
sudo nidstool up <SERVICE NAME>

Creates the sensor container (if it does not exist) and starts it. The Passive Sensor runs as a background task.

To start a sensor named nids-first, use:
sudo nidstool up nids-first
To start all sensors, use:
sudo nidstool up all
RUN

sudo nidstool run [--entrypoint=<BIN PATH>] <SERVICE NAME>
[<SENSOR ARGUMENTS>]

Runs the Passive Sensor container in interactive mode. The STDIN and STDOUT of the sensor are attached to the terminal that executed this command. Can be used to troubleshoot individual sensors.

  • SERVICE NAME must be the name of an individual sensor service (”all” is not allowed)
  • SENSOR ARGUMENTS are passed to the sensor as startup arguments. They must be separated from the nidstool arguments with a double-dash (--) separator, as in:
    sudo nidstool run nids-first -- --no-sniff
  • The --entrypoint option is for overriding the entry point of the container. This allows to run a different binary in the container instead of the sensor. For example, to access the container shell, use:
    sudo nidstool run --entrypoint=/bin/bash nids-first
 
STOP
sudo nidstool stop <SERVICE NAME>
Stops the sensor container without removing it.To stop a sensor named nids-first, use:
sudo nidstool stop nids-first

To stop all sensors, use:

sudo nidstool stop all

RESTART
sudo nidstool restart <SERVICE NAME>
Restarts the sensor containerTo restart a sensor named nids-first, use:
sudo nidstool restart nids-first

To restart all sensors, use:

sudo nidstool restart all

DOWN
sudo nidstool down <SERVICE NAME>
Stops and removes the sensor containerTo remove a sensor named nids-first, use:
sudo nidstool down nids-first

To remove all sensors, use:

sudo nidstool down all

UPDATE Updates specific sensor parameters, provided by the CERTS sub-command. 
CERTS (sub-command)
sudo nidstool update certs <CERT FOLDER PATH>
Replaces, validates, and applies the certificates used by all Passive Sensors.

The following files must be present in the <CERT FOLDER PATH> folder:

  • sensor cert.pem
  • sensor cc.crt
  • sensor key.pem
  • dh.pem

Refer to the Custom X.509 certificates in eyeInspect documentation for more information on how to generate these files.

If the private key of the certificate is encrypted, you will be prompted to enter its passphrase. The passphrase will also be updated in the nids.conf files.

 
CONFIG Applies or exports a sensor configuration file, determined by the provided sub-commands 
EXPORT (sub-command)
sudo nidstool config export <CONFIG PATH>

Exports the current configuration to the nidstool configuration file format. This sub-command generates the configuration based on the contents from the sensor Docker compose file and the configuration from all sensors (nids.conf and rdkafka.conf). This subcommand generates a new file in the path specified in <CONFIG PATH>.

 
APPLY (sub-command)

sudo nidstool config apply [--permissive] [--skip-confirmation]
<CONFIG PATH>

Applies the configuration from the nidstool configuration file. By default, it checks that the number of sniffing threads does not exceed the number of CPU cores of the machine and prompts the user before applying the configuration.

  • CONFIG PATH must point to a nidstool configuration file.
  • The --permissive option displays a warning instead of an error if the number of threads exceeds the number of CPU cores on the machine.
  • The --skip-confirmation option disables the confirmation prompt when applying the configuration

When applying a new configuration, nidstool shows the generated sensor configuration files in the terminal before applying.

New Passive Sensors use the same certificates as the first sensor in the nidstool configuration file. If the first sensor doesn’t have a custom certificate, the default eyeInspect certificates are used.

When an existing Passive Sensor is removed upon applying a new configuration, nidstool will save a backup of this sensor’s state folder in ’/opt/nids-docker/backups/states/<DATE HOUR MS>/<SENSOR NAME>’.

PCAP files are not preserved. Only the last 5 state folder backups are saved. If the number of backups exceeds this value, nidstool will ask for confirmation to remove the oldest backup folders.

Since this is a destructive operation, you need to confirm this operation even if the ”–skip-confirmation” option is used.

Applying a new configuration will remove all existing sensor containers. Sensors which were running before applying the configuration will be restarted, while new sensors must be manually started.
