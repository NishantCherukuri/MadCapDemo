Events Settings

The following events settings are available:

Event types

The Event type identifies what type of event was detected and by which type of detection profile. For example, the event type lan_cp_cnw_d shows that the detection profile that fired the alert is a LAN Connection Profiler. The cnw stands for connection not whitelisted, while the d denotes destination. This event type describes that the destination host is not whitelisted in any of the LAN CP communication rules.

Event types for Alerts can be customized, i.e., the name and description that is displayed with an Alert can be changed as desired, and new event types can be added, e.g., for SD Scripts. It is good practice to use the same format when adding new event types for SD Scripts and prefix all event types with "script".

The Event types page shows the predefined and customized event types. It is possible to:

  • Export/download the event types
  • Add a new event type
  • Import customized event types CSV

Advanced alert customization rules

Each eyeInspect Alert has an associated severity and status. In the default eyeInspect configuration, the severity depends on the Event type, and all Alerts first appear in the Alerts table as "Not analyzed". Users can change the default severity and status from the Advanced Alert Customization Rules page. Customization rules are named rules that set the severity and status of Alerts to the desired value based on any Alert attribute, e.g. source and destination IP, Event type, or protocol. This can be useful, for instance, to increase the severity of Alerts targeting critical hosts, or to mark Alerts originating from known scanners as false alerts.

Alert forwarding

The alert forwarding page allows users to enable and configure the forwarding of Alerts to a third-party solution by means of syslog messages. Alternatively, Alerts may be forwarded via e-mail. Each Alert may result in a distinct syslog or e-mail message.

Syslog forwarding

After filling in the basic information of a new syslog server, the connection details, header structure and message text can be set. Pre-set messages are available for CEF, LEEF and JSON format. Additionally, conditions for whether the Alert should be forwarded can be provided in the "Forwarding conditions" panel.

Example: Forwarding Alerts to an ArcSight Logger server using CEF

This section describes the procedure to forward Alerts to an ArcSight Logger via syslog using the Common Event Format (CEF) [1] . The configuration presented in this Section is for illustrative purposes only and was tested with ArcSight Logger version 5.3.0. Different versions of ArcSight Logger may require other adjustments to work.

The procedure is as follows:
  1. Go to "Settings" > "System Settings" > "Alert forwarding"
  2. Click on the plus (+) in the top-right corner of the syslog server table
  3. Enter the name and description
  4. Select "Finish"
  5. For the connectivity options enter:
    1. Transport protocolChoose UDP
      Remote hostThe IP address of your ArcSight Logger server
      Port NumberThe port number your ArcSight CEF
  6. For the header options leave the checkbox "Send message header" unchecked
  7. For the syslog message select the pre-set message CEF or type in the following text:
    1. CEF:0—Forescout—eyeInspect—–{sdVersion}—–{typeId}—–{name}—–{severityNum}—cat=alert externalId={ alertId} rt={tsFormatted} smac={srcMac} dmac={dstMac} src={srcIp} dst={dstIp} spt={srcPort} dpt={dstPort} proto={l4Proto} app={l7Proto} in={upDataLength} out={downDataLength} deviceDirection={streamDir} deviceExternalId={sensorName} cs1Label=ProfileModule cs1={profModName} cs2Label=FEAState cs2={feaState} cn1Label=AggrAlerts cn1={feaAlertCount} deviceCustomDate1Label=FEAStart deviceCustomDate1={feaStartMillisec} cn2Label= FEADurationSec cn2={feaDurationSec} cs3Label=FieldPath cs3={fieldPath} cs4Label=FieldVal cs4={fieldVal} cs5Label=ExpVals cs5={expFieldVals} cs6Label=Labels cs6={labels} msg={desc}
  8. For the message options set the following:
    1. Truncate large messagesLeave unchecked
      Replace newlines in rendered tags withLeave unchanged
      Replace carriage returns in rendered tags withLeave unchanged
      Replace/WithAdd a rule for replacing ”\” with ”\\” and a rule for replacing ”=” with ”\=”.
  9. Leave the forwarding condition as default with the "always active" checkbox ticked. If a customized forwarding condition is required un check the option and add a condition, for example, to forward only the alerts from a specific sensor.

E-mail forwarding

Configuring alert forwarding via e-mail is similar to setting up a Syslog forwarder. After filling in the basic information of a new e-mail server, the connectivity info can be set. This includes: the e-mail server address and port; the from address and sender name; the recipient list; security settings; and authentication information.

After this information has been filled in, you can click the "Test connectivity" button to verify the connection with the server can be established successfully. Optionally, alert forwarding conditions can be defined.

When setting up e-mail forwarding to users, it is advised that the operator checks user permissions of recipients. eyeInspect cannot determine if a recipient has view rights based on their e-mail address.

Network logs forwarding

The Network logs forwarding page ("Settings" > "System Settings" > "Network logs forwarding") allows users to enable and configure the forwarding of Network Logs to a third-party solution by means of syslog messages. The pages and configuration steps required to enable forwarding of Network Logs are exactly the same as those described for Alerts. The only difference lies in the semantics adopted when users uncheck the "always active" checkbox in the alert forwarding conditions, but leave the conditions "tree" empty. For Alerts, this results in all alerts being forwarded, whereas for Network Logs, this results in no log begin forwarded. The rationale is that Alerts are important events that are generally desirable to be forwarded to an analyst, whereas Network Logs are useful additional intelligence for context and threat hunting. This choice of default behavior is to prevent user mistakes in the configuration of eyeInspect to impact their monitoring capabilities. Pre-set messages for CEF, LEEF and JSON (Splunk) are available also for Network Logs forwarding.

Asset lifecycle events customization

Use this setting to raise an alert when an asset matches any of the defined conditions. Use the toggle at the top of the page to turn the alert feature on or off.

eyeInspect can create the following asset lifecycle events - use the toggle in each event type tile to turn the alerts on or off:

Security Risk Alerts

Users can choose to create an alert when an asset has risk equal to or higher than the level set on the asset risk level slider. Possible options are LOW, MEDIUM, HIGH, CRITICAL.

Operational Risk Alerts

Users can choose to create an alert when an asset has risk equal to or higher than the level set on the asset risk level slider. Available options on the slider are LOW, MEDIUM, HIGH, CRITICAL.

Risk alerts are triggered only if the asset's risk score was below the specified threshold prior to the calculation and exceeds it afterwards.
For example, if the operational risk alert threshold is set from medium to critical, the following applies:

Asset Vulnerability Alerts

To create an asset vulnerability alert:

  • Select one or more of the following vulnerabilities:
    • New
    • Updated
    • Suppressed
    • Resolved
  • Set the severity threshold using the slider. Options are: LOW, MEDIUM, HIGH, CRITICAL
  • Set the matching confidence threshold using the slider. Available options are POSS, PROB, CERT. Matching confidence represents how certain eyeInspect is that a vulnerability applies to an asset. It is calculated based on the amount of information known for a host or available on the vulnerability advisory.

Alerts will only be triggered if:

  • at least one vulnerability checkbox is selected

    AND

  • either a severity or matching threshold criteria is met
Asset Changelog Alerts

Users can choose to create an alert when an asset has a new host.
